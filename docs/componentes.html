Entregable 1 — DDD y Patrones de Diseno

Guia de componentes del repositorio. Logistica de ultima milla — Universidad de Valparaiso, 2026.

Problema: Una empresa de logistica de ultima milla opera sobre sistemas fragmentados desarrollados por areas independientes (ventas, operaciones, postventa, soporte). Esto genera inconsistencias en el ciclo de vida del pedido, dificultades de coordinacion y estados contradictorios entre sistemas. Se requiere modelar el dominio con DDD, aplicar patrones de diseno y principios SOLID para producir una arquitectura coherente.

Vision General de la Arquitectura

El sistema se estructura como 4 Bounded Contexts implementados (de un total de 6 modelados), siguiendo una arquitectura hexagonal por cada contexto: domain/ contiene las entidades, value objects y reglas de negocio; application/ orquesta los casos de uso; infrastructure/ provee las implementaciones concretas (repositorios en memoria).

Patrones Creacionales

Factory Method, Builder, Singleton

factory.py, builder.py, pool.py

Patrones Estructurales

Adapter, Facade, Decorator

adapter.py, despacho_service.py, validador.py

Patrones de Comportamiento

State, Strategy, Observer

estados.py, estrategias.py, domain_event.py

DDD Tactico

Aggregates, Value Objects, Domain Events, Repositories

pedido.py, incidencia.py, value_objects.py

Shared Kernel — src/shared/

Codigo compartido entre todos los bounded contexts. Contiene los building blocks fundamentales del dominio.

shared/value_objects.py Value Object DDD

Que hace: Define Direccion y ContactInfo, dos Value Objects inmutables (frozen=True) que representan conceptos del dominio sin identidad propia. Son compartidos por todos los bounded contexts que necesitan direcciones o datos de contacto.

Problema que resuelve: En el enunciado, cada pedido requiere direccion de origen, direccion de destino, nombre del destinatario y medio de contacto como informacion minima. Estos Value Objects encapsulan esa informacion con validaciones propias (es_valida(), tiene_medio_contacto()), evitando que la logica de validacion se disperse por el sistema.

@dataclass(frozen=True)
class Direccion:
    calle: str
    numero: str
    ciudad: str
    comuna: str

    def es_valida(self) -> bool:
        return all([self.calle, self.numero, self.ciudad, self.comuna])
shared/domain_event.py Observer Domain Event

Que hace: Define la clase base DomainEvent (inmutable, con ID unico y timestamp) y el EventBus, que implementa el patron Observer: permite suscribir handlers a tipos de eventos y publicar eventos de forma desacoplada.

Problema que resuelve: Cuando un pedido cambia de estado, multiples areas del negocio necesitan reaccionar (notificar al cliente, registrar en tracking, alertar a soporte). El Observer desacopla al emisor del evento de todos los suscriptores, cumpliendo con SRP (el pedido no necesita saber quien escucha) y permitiendo agregar nuevos suscriptores sin modificar el codigo existente (OCP).

class EventBus:
    def subscribe(self, event_type: type, handler: callable) -> None:
        if event_type not in self._subscribers:
            self._subscribers[event_type] = []
        self._subscribers[event_type].append(handler)

    def publish(self, event: DomainEvent) -> None:
        for handler in self._subscribers.get(type(event), []):
            handler(event)

BC: Gestion de Pedidos — src/pedidos/

Bounded Context core. Gestiona la creacion, validacion y ciclo de vida completo del pedido. Es el contexto mas rico en patrones de diseno.

pedidos/domain/value_objects.py Value Object

Que hace: Define tres enumeraciones del dominio: TipoEntrega (normal, express, programada), TipoCarga (estandar, fragil, voluminoso) y CanalOrigen (web, app_movil, ecommerce, partner).

Problema que resuelve: Las reglas de negocio del enunciado distinguen tipos de entrega con comportamientos distintos (una entrega programada requiere ventana de tiempo, una express tiene limite de peso). Los enums garantizan que solo se usen valores validos del dominio, evitando strings sueltos.

pedidos/domain/estados.py State LSP

Que hace: Implementa el patron State para los 9 estados del pedido: Creado, Validado, Pendiente de asignacion, Asignado, En ruta, Intento fallido, Reprogramado, Entregado y Cancelado. La clase abstracta EstadoPedido define todas las operaciones posibles; cada estado concreto sobreescribe solo las transiciones permitidas y lanza TransicionInvalidaError para las demas.

Problema que resuelve: El enunciado define un flujo no lineal donde "un pedido no puede ser asignado si no esta validado", "un pedido entregado no puede volver a estados anteriores", y "un pedido cancelado no puede continuar el flujo". Sin State, esto requeriria cadenas interminables de if/elif. Con State, cada estado sabe exactamente que transiciones permite, y la maquina de estados se auto-documenta. LSP: todos los estados son intercambiables via la interfaz EstadoPedido.

class Asignado(EstadoPedido):
    def iniciar_ruta(self, pedido):
        pedido.cambiar_estado(EnRuta())       # Transicion valida

    def poner_pendiente(self, pedido):
        pedido.cambiar_estado(PendienteAsignacion())  # Reasignacion

class Entregado(EstadoPedido):
    # Estado final: no sobreescribe ningun metodo
    # Cualquier operacion lanza TransicionInvalidaError
    def nombre(self): return "Entregado"
pedidos/domain/pedido.py Aggregate Root State SRP

Que hace: Define Pedido como Aggregate Root del bounded context. Contiene toda la informacion requerida por las reglas de negocio (origen, destino, destinatario, tipo entrega, carga, canal) y delega cada operacion de cambio de estado al objeto EstadoPedido actual. Tambien acumula eventos de dominio para publicacion posterior.

Problema que resuelve: Pedido es la entidad central del dominio; todo gira en torno a el. Al delegarle las transiciones al State, la clase Pedido se mantiene simple (SRP: solo modela la entidad, no decide reglas de transicion). Al ser Aggregate Root, es el punto de entrada para cualquier modificacion dentro de este contexto — ningun objeto interno se modifica sin pasar por el.

def asignar(self, repartidor_id: str) -> None:
    self._estado.asignar(self)   # Delega al State
    self.repartidor_id = repartidor_id
pedidos/domain/builder.py Builder

Que hace: Implementa el patron Builder para la construccion paso a paso de objetos Pedido. Expone metodos encadenables (con_canal(), con_origen(), con_destino(), con_entrega(), con_carga()) y valida la completitud antes de construir el objeto final con build().

Problema que resuelve: El enunciado exige que un pedido tenga como minimo: direccion de retiro, punto de origen, direccion de destino, nombre destinatario, medio de contacto, tipo de entrega, ventana de tiempo (si aplica), tipo de carga, peso y canal de origen. Son muchos campos, algunos condicionales. Un constructor con 10+ parametros seria ilegible. Builder permite construir el pedido de forma gradual, validando al final que no falte informacion critica.

pedido = (PedidoBuilder()
    .con_canal(CanalOrigen.WEB)
    .con_origen(direccion_bod, "BOD-001")
    .con_destino(direccion_cliente, contacto)
    .con_entrega(TipoEntrega.EXPRESS)
    .con_carga(TipoCarga.ESTANDAR, peso_kg=5.0)
    .build())
pedidos/domain/factory.py Factory Method OCP

Que hace: Implementa el patron Factory Method con una clase abstracta PedidoFactory y cuatro subclases concretas: WebPedidoFactory, AppMovilPedidoFactory, EcommercePedidoFactory y PartnerPedidoFactory. La funcion obtener_factory(canal) retorna la fabrica correcta segun el canal.

Problema que resuelve: El enunciado dice que "un pedido puede ser creado con distintos niveles de informacion" desde "diversos canales". Cada canal podria tener datos, formatos o validaciones distintas. Factory Method permite que agregar un nuevo canal (ej: TelefonoPedidoFactory) sea crear una nueva clase sin tocar ninguna fabrica existente — cumpliendo OCP directamente.

class PedidoFactory(ABC):
    @abstractmethod
    def crear_pedido(self, datos: dict) -> Pedido: ...

class WebPedidoFactory(PedidoFactory):
    def crear_pedido(self, datos):
        return self._builder_base(datos, CanalOrigen.WEB).build()

class PartnerPedidoFactory(PedidoFactory):   # Nuevo canal: 0 cambios en las demas
    def crear_pedido(self, datos):
        return self._builder_base(datos, CanalOrigen.PARTNER).build()
pedidos/domain/validador.py Decorator OCP ISP

Que hace: Implementa el patron Decorator para apilar capas de validacion sobre pedidos. ValidadorBase verifica la informacion minima. Los decoradores ValidadorExpress, ValidadorProgramada y ValidadorZona agregan reglas adicionales sin modificar la validacion base. Se componen con crear_validador_completo().

Problema que resuelve: Las reglas de negocio del enunciado dicen que "la validacion puede depender de condiciones operativas" y que hay restricciones especificas por tipo de entrega (peso para express, ventana para programada). Sin Decorator, cada nueva regla obligaria a modificar un unico metodo validar() gigante. Con Decorator, cada regla vive en su propia clase (SRP), se apilan libremente (OCP: agregar ValidadorFragil no modifica nada existente), y la interfaz es minima: un solo metodo validar() (ISP).

# Composicion de validadores (Decorator chain)
validador = ValidadorBase()
validador = ValidadorExpress(validador)      # Agrega regla: peso < 20kg para express
validador = ValidadorProgramada(validador)   # Agrega regla: ventana obligatoria
validador = ValidadorZona(validador)         # Agrega regla: ciudad atendida
pedidos/domain/repository.py DIP Repository

Que hace: Define la interfaz abstracta PedidoRepository con los metodos guardar(), obtener_por_id() y listar_todos().

Problema que resuelve: DIP en estado puro. Los servicios de aplicacion dependen de esta abstraccion, no de la implementacion concreta. Si manana se reemplaza el repositorio en memoria por PostgreSQL, solo cambia la implementacion en infrastructure/; ni PedidoService ni el dominio se modifican.

pedidos/infrastructure/in_memory_repo.py Infrastructure DIP

Que hace: Implementa PedidoRepository usando un diccionario en memoria. Cumple el requisito del enunciado de "no implementar nada en motor de base de datos, solo simulacion desde codigo".

pedidos/application/pedido_service.py SRP DIP

Que hace: Servicio de aplicacion que orquesta el caso de uso Gestion de Pedidos. Coordina la creacion (via Factory), validacion (via Decorator), transiciones de estado (via State) y publicacion de eventos (via Observer). Recibe sus dependencias por constructor (inyeccion de dependencias).

Problema que resuelve: SRP: no implementa logica — solo orquesta las piezas del dominio. DIP: recibe PedidoRepository (abstraccion), ValidadorPedido (abstraccion) y EventBus, nunca implementaciones concretas.

class PedidoService:
    def __init__(self, repository: PedidoRepository,  # abstraccion
                 event_bus: EventBus,
                 validador: ValidadorPedido | None = None):
        ...

    def crear_pedido(self, canal: str, datos: dict) -> Pedido:
        factory = obtener_factory(canal)     # Factory Method
        pedido = factory.crear_pedido(datos)  # Builder internamente
        self._repository.guardar(pedido)
        self._event_bus.publish(...)           # Observer
        return pedido

BC: Gestion de Repartidores — src/repartidores/

Bounded Context core. Gestiona el registro, disponibilidad y capacidad de la flota de repartidores.

repartidores/domain/repartidor.py Entity

Que hace: Entidad Repartidor con identidad propia, que encapsula las reglas de negocio: debe estar disponible para recibir pedidos, no puede superar su capacidad, y puede manejar multiples pedidos simultaneamente.

Problema que resuelve: El enunciado establece restricciones claras sobre repartidores. La entidad valida estas reglas internamente: asignar_pedido() lanza error si no esta disponible o si se excede la capacidad, garantizando consistencia sin depender de validaciones externas.

def asignar_pedido(self, pedido_id: str) -> None:
    if not self.disponible:
        raise ValueError(f"Repartidor {self.nombre} no esta disponible")
    if not self.tiene_capacidad:
        raise ValueError(f"Capacidad maxima alcanzada ({self.capacidad.max_pedidos})")
repartidores/domain/pool.py Singleton

Que hace: Implementa el patron Singleton para DriverPool, un pool centralizado que administra la flota completa de repartidores disponibles. Sobreescribe __new__ para garantizar una unica instancia.

Problema que resuelve: El despacho necesita una fuente de verdad unica sobre que repartidores estan disponibles. Si existieran multiples pools, la asignacion podria asignar al mismo repartidor a dos pedidos distintos. Singleton garantiza que todos los servicios consultan el mismo pool, manteniendo consistencia global.

class DriverPool:
    _instance: DriverPool | None = None

    def __new__(cls) -> DriverPool:
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance._repartidores = {}
        return cls._instance
repartidores/application/repartidor_service.py SRP DIP

Que hace: Servicio de aplicacion para el caso de uso Gestion de Repartidores. Orquesta registro, cambio de disponibilidad, actualizacion de ubicacion y consulta del pool.

Problema que resuelve: Misma separacion que PedidoService: orquesta sin implementar logica. Depende de RepartidorRepository (abstraccion).

BC: Asignacion y Despacho — src/despacho/

Bounded Context core. Orquesta la asignacion de pedidos a repartidores, aplicando estrategias intercambiables y encapsulando la complejidad tras una fachada.

despacho/domain/estrategias.py Strategy OCP LSP

Que hace: Implementa el patron Strategy con tres algoritmos de asignacion intercambiables: AsignacionPorProximidad (elige el repartidor mas cercano), AsignacionPorCapacidad (elige el de mayor capacidad disponible) y AsignacionRoundRobin (distribuye equitativamente). Todos implementan la interfaz EstrategiaAsignacion.

Problema que resuelve: El enunciado habla de "asignacion automatica" y "re-asignacion", con condiciones del entorno que impactan la ejecucion. Distintos momentos pueden requerir distintos criterios. Strategy permite cambiar el algoritmo en tiempo de ejecucion sin modificar el servicio (OCP). Todas las estrategias son intercambiables sin romper el contrato (LSP).

class EstrategiaAsignacion(ABC):
    @abstractmethod
    def seleccionar(self, disponibles, ubicacion_destino) -> Repartidor | None: ...

class AsignacionPorProximidad(EstrategiaAsignacion):
    def seleccionar(self, disponibles, ubicacion_destino):
        return min(disponibles, key=lambda r: r.ubicacion.distancia_a(ubicacion_destino))
despacho/application/adapter.py Adapter

Que hace: Implementa el patron Adapter para traducir entidades de otros bounded contexts al modelo interno de Despacho. PedidoAdapter convierte un Pedido en PedidoParaDespacho; RepartidorAdapter convierte un Repartidor en RepartidorParaDespacho.

Problema que resuelve: El enunciado menciona que "cada canal fue desarrollado por equipos diferentes" y los sistemas tienen interfaces distintas. El Adapter protege al contexto de Despacho de los detalles internos de Pedidos y Repartidores — si el modelo de Pedido cambia, solo se actualiza el Adapter, no todo el subsistema de despacho. Actua como un Anti-Corruption Layer (ACL) a nivel tactico.

class PedidoAdapter:
    @staticmethod
    def adaptar(pedido: Pedido) -> PedidoParaDespacho:
        return PedidoParaDespacho(
            pedido_id=pedido.pedido_id,
            peso_kg=pedido.peso_estimado_kg,
            tipo_entrega=pedido.tipo_entrega.value,
            estado=pedido.estado_nombre, ...)
despacho/application/despacho_service.py Facade Strategy SRP

Que hace: Implementa el patron Facade. DespachoService expone metodos simples (asignar_pedido(), reasignar_pedido()) que internamente orquestan 5 pasos: obtener pedido, verificar estado, buscar repartidores disponibles en el pool, aplicar la estrategia de seleccion, asignar y publicar evento. Tambien permite cambiar_estrategia() en runtime.

Problema que resuelve: El proceso de despacho toca multiples subsistemas (pedidos, repartidores, asignaciones, eventos). Sin Facade, el cliente tendria que coordinar todo esto manualmente. La Facade oculta esa complejidad: quien llama solo necesita despacho_service.asignar_pedido(id).

def asignar_pedido(self, pedido_id: str) -> Asignacion:
    # 1. Obtiene pedido y verifica estado
    # 2. Busca repartidores disponibles en el pool (Singleton)
    # 3. Aplica estrategia de seleccion (Strategy)
    # 4. Asigna pedido al repartidor
    # 5. Publica evento de dominio (Observer)
    repartidor = self._estrategia.seleccionar(disponibles, ubicacion)
    repartidor.asignar_pedido(pedido_id)
    pedido.asignar(repartidor.repartidor_id)
    ...

BC: Gestion de Incidencias — src/incidencias/

Bounded Context generico. Gestiona reclamos, investigacion y resolucion de eventos anomalos sobre pedidos.

incidencias/domain/estados.py State

Que hace: Implementa el patron State para el ciclo de vida de incidencias con 4 estados: Abierta, En analisis, En resolucion y Resuelta. La transicion a "Resuelta" valida que la incidencia tenga una resolucion asignada.

Problema que resuelve: El enunciado define que "una incidencia no puede cerrarse sin resolucion". El estado EnResolucion valida esto explicitamente: si no hay resolucion asignada, lanza un error. Esto garantiza la regla de negocio a nivel de dominio, no como validacion externa.

class EnResolucion(EstadoIncidencia):
    def resolver(self, incidencia):
        if not incidencia.resolucion:
            raise ValueError("No se puede cerrar sin resolucion asociada")
        incidencia.cambiar_estado(Resuelta())
incidencias/domain/incidencia.py Aggregate Root State

Que hace: Aggregate Root Incidencia que referencia un pedido por ID (no por objeto — respetando limites del bounded context). Contiene tipo, descripcion, resolucion y estado. Delega operaciones al State.

Problema que resuelve: El enunciado dice que "toda incidencia debe estar asociada a un pedido" y que "puede gatillar nuevas acciones sobre el pedido". La incidencia referencia al pedido por ID y su Resolucion (Value Object) puede indicar requiere_reenvio=True, permitiendo que otros servicios reaccionen.

incidencias/application/incidencia_service.py SRP DIP

Que hace: Servicio de aplicacion para el caso de uso Gestion de Incidencias. Permite registrar incidencias (verificando que el pedido exista), avanzar estados y asignar resoluciones.

Problema que resuelve: Orquesta las operaciones sin implementar logica de dominio (SRP). Depende de IncidenciaRepository y PedidoRepository como abstracciones (DIP), comunicandose con el BC de Pedidos solo a traves de su interfaz de repositorio.

Principios SOLID — Mapa Completo

S — Single Responsibility

Cada clase tiene una unica razon para cambiar. Pedido modela, ValidadorBase valida, PedidoRepository persiste, PedidoService orquesta, PedidoBuilder construye, EstadoPedido gestiona transiciones. Ninguna clase mezcla responsabilidades.

O — Open/Closed

El sistema es extensible sin modificaciones. Nuevo canal: crear una PedidoFactory. Nueva estrategia: crear una EstrategiaAsignacion. Nueva validacion: crear un ValidadorDecorator. Nuevo suscriptor a eventos: llamar a event_bus.subscribe(). Cero cambios en codigo existente.

L — Liskov Substitution

Todas las subclases son intercambiables con sus clases base. Los 9 estados del pedido implementan EstadoPedido. Las 3 estrategias implementan EstrategiaAsignacion. Las 4 fabricas implementan PedidoFactory. Los decoradores implementan ValidadorPedido. Ninguna rompe el contrato de su interfaz.

I — Interface Segregation

Las interfaces son minimas y especificas. ValidadorPedido tiene un solo metodo: validar(). PedidoRepository tiene 3 metodos de persistencia. EstrategiaAsignacion tiene seleccionar() y nombre(). No hay interfaces monoliticas que fuercen a implementar metodos innecesarios.

D — Dependency Inversion

Los servicios de aplicacion dependen de abstracciones. PedidoService recibe PedidoRepository (ABC), no InMemoryPedidoRepository. DespachoService recibe EstrategiaAsignacion (ABC), no AsignacionPorProximidad. La inyeccion ocurre en main.py — el unico lugar que conoce las implementaciones concretas.

Context Map y Relaciones entre Bounded Contexts

El dominio completo modela 6 bounded contexts (4 implementados). Las relaciones se definen asi:

Pedidos → Despacho — Customer/Supplier: Pedidos provee datos para que Despacho asigne repartidores.
Repartidores → Despacho — Customer/Supplier: Repartidores provee disponibilidad y capacidad.
Despacho → Rutas — Customer/Supplier: Despacho alimenta las rutas con pedidos asignados.
Pedidos → Tracking — Conformist: Tracking se adapta completamente al modelo de Pedidos.
Pedidos → Incidencias — Customer/Supplier: Incidencias referencia pedidos por ID.
Tracking → Notificaciones — ACL (Anti-Corruption Layer): protege el dominio de sistemas externos.
Pedidos → E-commerce / Partners — ACL: Adapter traduce formatos de canales externos.
